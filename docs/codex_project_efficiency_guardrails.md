# Codex 通用高效开发准则

适用于所有本地项目、Web/App 项目、前后端联调、局域网展示、ngrok 公网展示、GitHub 发布和服务排查工作。

## 核心目标

减少三类问题：

- 看起来卡住：命令无输出、接口太大、网络等待、进程挂起。
- 改了没生效：构建目标错、服务没重启、访问的是旧入口。
- 排查方向错：把 ngrok 问题当应用问题，把权限问题当代码问题，把本地端口问题当公网问题。

最终目标：

- 路径清楚；
- 目标清楚；
- 验证轻量；
- 输出及时；
- 改动可回滚；
- 用户随时知道当前进度。

## 启动前检查

每次进入项目，先做快速确认：

1. 当前路径：确认 `pwd`，避免改错副本或软链接目标。
2. Git 状态：查看 `git status --short`，确认是否有用户未提交改动。
3. 最近提交：查看 `git log --oneline -3`，确认当前版本位置。
4. 访问入口：明确本次验证用本地、局域网、公网、GitHub Pages 还是生产环境。
5. 服务层级：区分前端、后端、网关/代理、数据库、公网隧道。

## 命令执行准则

- 所有网络命令必须加时间上限，`curl` 默认使用 `--max-time 5` 或 `--max-time 10`。
- 不用完整 Dashboard、大 JSON、大日志作为首次验证。
- 优先使用状态码、`/health`、小接口、单字段检查。
- `sed`、`rg`、`tail` 都限制范围，避免一次性打印大量内容。
- 并行只用于轻量读取，不并行运行长构建、服务启动、push、大接口请求。
- 如果命令可能长时间运行，先说明正在等什么。
- 超过约 30 秒没有明显进展，要主动给用户状态更新。

## 权限准则

普通权限失败一次即可。遇到以下情况，直接申请提升权限，不反复试：

- `Operation not permitted`
- 端口绑定失败；
- `.git/index.lock` 无法创建；
- macOS 网络配置不可读或不可改；
- 进程无法停止或重启；
- 必须联网下载、同步或验证。

提升权限必须说明目的，并限制在当前动作范围内。

## 服务分层排查

页面、登录、API、局域网、公网打不开时，按层排查：

1. 静态页面是否返回状态码；
2. 前端资源路径是否正确；
3. 后端健康接口或小接口是否正常；
4. 网关/代理是否转发到正确目标；
5. 数据库或本地数据文件是否可用；
6. LAN IP、端口监听、防火墙是否正确；
7. ngrok 或公网隧道是否在线；
8. 浏览器缓存或旧构建是否影响。

不要把公网隧道失败当成应用失败，也不要在没有定位层级前修改业务代码。

## 构建与重启准则

- 每个项目记录实际展示用的构建命令。
- 区分本地开发命令和局域网/公网/静态展示构建命令。
- 前端改动后，要构建最终访问入口使用的版本。
- 后端路由、字段、数据库逻辑改变后，必须重启后端。
- 配置、路径、base path 改动后，必须验证最终 URL。

## 验证顺序

推荐顺序：

1. 语法、类型或构建检查；
2. 重启变更过的服务；
3. 小 API 或状态码探测；
4. 用户真实访问入口检查；
5. Git 状态和聚焦 diff；
6. 提交和推送。

先小验证，再扩大验证范围。

## Git 与发布准则

- 只暂存本次相关文件。
- 不提交缓存、日志、临时文件和无关元数据。
- commit message 要说明真实用户可感知改动。
- 推送失败如疑似代理问题，优先使用：

```bash
git -c http.proxy= -c https.proxy= push
```

- 推送后报告 commit hash 和远端是否成功。

## 前端交互检查

所有“看起来像按钮”的元素必须：

- 可点击并产生行为；
- 或明确显示为禁用、规划中、不可用。

检查项：

- 导航能跳转、滚动、展开或给反馈；
- Tab 必须切换内容；
- 表单操作有成功/失败反馈；
- 版本号和实际功能一致；
- 移动端无横向溢出和文字重叠；
- 最终 URL 使用最新构建。

## 沟通准则

- 长任务中要短频更新。
- 用户说“卡住了”“思考太久”时，立即停止宽泛排查，先汇报状态。
- 不说“还在思考”这类空话，要说明正在等哪条命令或哪个服务层。
- 如果验证路径很慢，立刻换成更小的检查。
- 如果判断层级错了，明确说明更正后的原因。

## 持续迭代

每次出现新的卡点，补充到准则里：

- 用户看到的症状；
- 实际原因；
- 下次最快检测方法；
- 推荐修复方式；
- 应避免的命令或流程。

一句话原则：

```text
先定位层级，再小步验证；先短命令，再长任务；先让用户知道进度，再继续执行。
```
