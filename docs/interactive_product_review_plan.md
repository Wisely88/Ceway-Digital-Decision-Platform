## 当前共性问题

### 1. 功能能跑，但产品任务流不够强

两个系统都有模块、数据和接口，但用户进入后还需要自己推理：

- 我现在在哪个业务阶段？
- 当前数据是不是最新？
- 点这个按钮之后结果在哪里？
- 保存了以后去哪里找？
- 这个推荐/分析和历史结果如何闭环？

这会让系统像“开发完成的功能集合”，不像“可日常使用的工具”。

### 2. 交互反馈不足

典型问题：

- 点击生成后不知道是否成功；
- 点击保存后没有明确目标页；
- 失败时没有区分输入错误、后端错误、网关错误、数据错误；
- 移动端操作后缺少状态提示。

修复原则：

所有动作都要有 4 种状态：

- 待操作；
- 执行中；
- 成功；
- 失败与恢复路径。

### 3. 模块导航曾经只是页面跳转

CBGO 已开始改成模块切换，但后续仍要统一为：

- 左侧或顶部模块导航；
- 当前模块明确高亮；
- 模块内容独立；
- 不让用户滚完整个系统。

ClawScore 也应避免“所有数据堆在一屏”，改为比赛列表、比赛详情、信号对比、历史记录等任务页。

## CBGO 体验差异盘点

### 当前用户真实任务

1. 进入场景中心。
2. 选择大乐透或双色球。
3. 确认最新开奖期号和数据来源。
4. 输入预算与策略。
5. 生成推荐方案。
6. 理解为什么生成这组号码。
7. 保存方案。
8. 开奖后做推荐与开奖对比。
9. 查看历史回测和资金状态。

### 当前主要差距

#### P0

1. **生成与保存反馈必须继续强化**
   - 已修：生成后切到组合生成模块。
   - 已修：保存中状态与保存后跳历史记录。
   - 仍需补：保存失败时把错误分层显示为“后端保存失败 / 本地浏览器已暂存”。

2. **预算解释不够产品化**
   - 当前能做到预算内生成，但用户会疑惑为什么 50 元只生成 42 元或 40 元。
   - 应增加“预算匹配说明”：
     - 当前预算；
     - 可用最大合法费用；
     - 剩余未使用金额；
     - 原因：胆拖费用由组合公式决定，不一定刚好等于预算。

3. **复盘链路需要放到更显眼位置**
   - 用户保存方案后，下一期应该能直接看到：
     - 推荐期号；
     - 实际开奖；
     - 前区/红球命中；
     - 后区/蓝球命中；
     - 是否中奖；
     - 资金影响。

#### P1

4. **组合生成模块需要更像业务工具**
   - 增加方案列表而不是只看一个卡片：
     - 推荐方案；
     - 备选方案 A；
     - 备选方案 B；
     - 单式兜底方案。
   - 每个方案显示：
     - 费用；
     - 注数；
     - 覆盖结构；
     - 预算占用率；
     - 生成理由。

5. **数据管理要变成状态中心**
   - 显示：
     - 最新期号；
     - 最新开奖日期；
     - 数据源；
     - 同步时间；
     - 缺失期数；
     - 手动导入入口。

6. **号码评分与趋势分析的连接不够明显**
   - 趋势页应支持点击号码，打开号码详情：
     - 热度；
     - 遗漏；
     - 评分；
     - 最近出现期号；
     - 是否进入推荐。

#### P2

7. **移动端需要专门工作流**
   - 场景页、控制区、组合生成、历史记录需要手机端单列任务流。
   - 侧栏在手机端应折叠为顶部模块切换或底部导航。

8. **页面文案还偏系统说明**
   - 应减少“系统在做什么”的说明，增加“用户下一步能做什么”。

## CBGO 推荐改造方案

### Phase 1：任务闭环修复

目标：

让用户完成一次完整任务：

选择场景 -> 确认数据 -> 设置预算 -> 生成 -> 保存 -> 复盘。

改动：

- 组合生成增加预算匹配说明；
- 保存后历史记录置顶并显示“待复盘”状态；
- 推荐复盘增加“下一步：等待第 X 期开奖”；
- 错误提示分层；
- 数据状态固定在总览页。

验收：

- 用户不用滚动长页面；
- 点击生成一定看到新方案；
- 点击保存一定看到保存结果；
- 预算未用满时有明确解释。

### Phase 2：业务专业感增强

目标：

让 CBGO 不像“随机选号页面”，而像“决策辅助工作台”。

改动：

- 方案列表支持主推/备选/单式；
- 评分表点击号码打开详情；
- 复盘页按推荐期号归档；
- 历史记录支持筛选：场景、期号、策略、命中状态；
- 资金管理从静态状态改成基于保存记录和开奖结果计算。

验收：

- 用户能解释“为什么这组号码被推荐”；
- 用户能查看每次推荐的后续结果；
- 资金状态不是装饰图，而是由真实记录驱动。

## ClawScore 体验差异盘点

### 当前用户真实任务

1. 打开系统。
2. 查看今日或指定日期比赛。
3. 筛选联赛、球队、状态。
4. 进入某场比赛详情。
5. 对比关键指标或信号。
6. 保存/标记关注比赛。
7. 回看结果和历史表现。

### 当前可能差距

#### P0

1. **入口应先服务“今日比赛”**
   - 足球数据用户打开系统通常先看今天有什么。
   - 首屏应优先显示：
     - 今日比赛数；
     - 进行中；
     - 未开赛；
     - 已完赛；
     - 数据更新时间。

2. **筛选器必须稳定**
   - 日期、联赛、球队、状态是核心筛选。
   - 筛选后要显示当前条件，不应让用户猜自己在看哪一组比赛。

3. **比赛详情要有明确结构**
   - 基本信息；
   - 近期走势；
   - 对阵数据；
   - 指标/信号；
   - 历史结果；
   - 用户备注或关注。

#### P1

4. **数据密集页要减少装饰卡片**
   - 足球工具更适合表格、列表、筛选、比较面板。
   - 不适合大量营销式卡片。

5. **重要信号需要解释**
   - 如果有评分、指数、预测、热度类字段，必须说明计算依据、更新时间和限制。

6. **关注列表/历史记录**
   - 用户需要回看自己关注过哪些比赛，以及结果如何。

#### P2

7. **移动端应优先比赛列表**
   - 手机端不要先展示大图表；
   - 应是日期切换 + 比赛列表 + 状态标签 + 快速进入详情。

## ClawScore 推荐改造方案

### Phase 1：比赛工作流重构

- 首屏改成“今日比赛工作台”；
- 日期条固定在顶部；
- 联赛/球队/状态筛选常驻；
- 比赛列表支持状态标签；
- 点击比赛进入详情页或右侧详情面板。

### Phase 2：详情页专业化

- 详情页分区：
  - 比赛信息；
  - 双方近期；
  - 历史交锋；
  - 指标对比；
  - 风险提示；
  - 关注/备注。

### Phase 3：复盘与记录

- 增加关注比赛记录；
- 赛后自动进入复盘状态；
- 展示赛前判断与赛后结果差异。

## 两个系统共同验收标准

每次交互改造必须通过这 8 条：

1. 用户第一屏知道当前场景和数据新鲜度。
2. 主要按钮点击后有 loading / success / error。
3. 生成或分析结果出现在用户预期位置。
4. 保存后能立即看到保存记录。
5. 错误信息能区分输入、数据、后端、网关、公网隧道。
6. 移动端不需要横向拖页面才能完成主任务。
7. 模块导航切换真实内容，不只是滚动锚点。
8. 每个系统都有“历史记录 -> 复盘结果”的闭环。

## 下一步建议

优先顺序：

1. CBGO Phase 1：补预算匹配说明、保存/复盘闭环、错误分层。
2. ClawScore Phase 1：今日比赛工作台、筛选器、比赛详情结构。
3. 两个系统都做移动端任务流检查。
4. 再考虑视觉精修和动效。

这套计划应作为后续 UI/UX 开发的准入标准：先任务流，再视觉；先可用，再漂亮。
