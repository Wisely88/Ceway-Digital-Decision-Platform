# 实用 UI / App 设计通用准则

适用于网页、App、Dashboard、SaaS 工具、数据产品、运营后台、场景中心、移动端页面和交互式工具开发。

核心目标：

- 界面贴近真实使用场景；
- 布局清晰美观；
- 交互直接有效；
- 模块化便于迭代；
- 尽量减少后续返工；
- 不为炫技牺牲实用性。

## 设计起点

开发界面前，先判断：

1. 用户是谁：操作员、管理者、创作者、消费者、分析师、管理员、休闲用户还是购买者。
2. 主要任务是什么：监控、决策、比较、创建、配置、复盘、购买、学习还是娱乐。
3. 使用频率：一次性、偶尔、日常、高频操作。
4. 信息密度：轻内容页、中等产品页、密集数据看板、专家工具。
5. 使用设备：桌面优先、移动优先、平板、共享屏、展厅屏。

界面首先服务任务，不默认套用营销式 Hero、大量装饰卡片、夸张渐变或过大标题。

## 排版布局准则

- 第一屏优先呈现主工作流。
- 用户要求的是工具或系统时，不做空洞落地页。
- 页面区域保持稳定：顶部信息、导航、控制区、主体内容、详情区、反馈区、底部说明。
- 导航层级匹配复杂度：
  - 1-3 个视图：Tab 或分段控制；
  - 多模块系统：侧边栏或顶部导航；
  - 深层详情：面包屑或明确返回按钮。
- Dashboard 推荐顺序：
  - 状态总览；
  - 参数控制；
  - 核心分析；
  - 明细；
  - 历史/日志。
- 控件靠近它影响的内容。
- 主要操作突出，次要操作克制。
- 不嵌套卡片。
- 卡片只用于重复项目、工具面板、记录、弹窗等真正需要边界的内容。

## 视觉层级准则

- 每页只保留一个主标题。
- 面板内部标题要紧凑。
- 不用视口宽度动态缩放字体。
- 不使用负字距。
- 颜色用于表达状态、类别、动作和重点，不只做装饰。
- 避免单一色系统治整个界面。
- 不默认使用大面积紫色、深蓝、米色、咖啡色或炫光渐变。
- 常见工具动作优先使用图标，重要动作图标加文字。
- 数字、状态、下一步要一眼可扫。
- 产品、地点、人物、真实数据要尽早出现，避免空泛氛围图。

## 交互准则

- 所有可见按钮必须有效，或明确显示禁用/规划中。
- Tab 必须切换实际内容。
- 导航必须跳转、滚动、展开或高亮反馈。
- 输入框要有标签、默认值、约束和错误反馈。
- 保存、复制、导出、生成等操作必须有成功/失败提示。
- 超过约 300ms 的操作要显示加载状态。
- 危险操作需要确认或撤销机制。
- 高频工具要照顾键盘和鼠标连续操作体验。

## 动态效果准则

动态效果用于解释状态，不用于炫技。

适合使用：

- hover / focus 微反馈；
- 加载/进度反馈；
- 面板展开收起；
- 图表更新过渡；
- 弹窗或页面切换的短动效。

避免：

- 工作型工具里持续背景动画；
- 与数据抢注意力的装饰动画；
- 默认使用视差、漂浮球、光斑、动态渐变；
- 导致布局跳动或文字重叠的动画。

动效时长应短，常规转场建议控制在 250ms 内。

## 移动端准则

- 移动端不是缩小版桌面，要重新组织结构。
- 侧边栏应折叠为顶部、底部导航或抽屉。
- 主要按钮高度不低于约 42px。
- 除数据表和图表外，避免横向滚动。
- 窄屏优先单列布局。
- 图表必要时使用稳定 min-width 横向滚动。
- 文本不能覆盖控件或卡片。
- 要测试最长中文标签，而不是只看理想短文本。

## 模块化设计准则

重复模式应抽象为组件：

- 统计卡片；
- 面板；
- 表格；
- Tab；
- 筛选器；
- 记录项；
- 表单；
- 空状态；
- 报错提示；
- 成功提示。

组件接口保持简单：

- 标题；
- 副标题；
- 操作；
- 状态；
- 内容区。

状态必须一开始就考虑：

- loading；
- empty；
- error；
- success；
- disabled；
- stale；
- updating。

只在真实减少复杂度或支持后续模块时增加抽象。

## 场景化设计默认策略

### SaaS / CRM / 运营后台

适合紧凑、冷静、高信息密度设计。优先表格、筛选、侧边详情、快捷操作、状态徽标、审计日志。避免大 Hero 和装饰性卡片堆叠。

### 数据分析 / Dashboard

时间范围、筛选条件、数据更新时间靠前。核心指标在图表之前。图表要配合解释、限制说明和可下钻明细。

### 消费产品 / 品牌 / 场馆页面

第一屏要看到真实产品、地点、人物或对象。图片要有辨识价值，不能只是模糊氛围。主要转化按钮保持可见。

### 创意作品 / Portfolio

可以更有表现力，但仍要保证导航清晰、文字可读、作品样本真实。

### 游戏 / 交互工具

第一屏直接呈现可玩或可操作区域。视觉风格可以更强，但必须服务理解和参与。

## 完工前 UI 检查清单

交付前必须检查：

1. 主工作流是否第一屏可用；
2. 所有可见控件是否有行为或明确状态；
3. 导航和 Tab 是否真的改变内容或位置；
4. 桌面和手机文本是否都能放下；
5. 是否存在不必要的卡片嵌套；
6. loading、empty、error、success 状态是否覆盖关键流程；
7. 版本和状态标签是否与已实现功能一致；
8. 最终展示 URL 是否加载最新构建。

## 审美原则

好的界面不是“元素很多”，而是：

- 用户知道自己在哪；
- 用户知道下一步做什么；
- 重要信息不需要找；
- 操作结果有反馈；
- 页面看起来克制、稳定、可信；
- 美观服务效率，而不是干扰效率。

一句话原则：

```text
能简洁就简洁，能直接就直接；先实用，再精致；先适合场景，再谈风格。
```
